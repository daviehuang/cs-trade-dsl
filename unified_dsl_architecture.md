# 统一业务规则 DSL —— 架构设计文档

> 状态：Draft v0.1
> 关联文档：[unified_dsl_summary.md](./unified_dsl_summary.md)（设计目标）
> 本文目标：把"概念阶段"的设计目标，细化为可落地的架构规范。重点细化四件事——
> **UI/Server 双执行**、**Server 最终裁决**、**Function 治理**、**规则版本化与父子作用域执行**。

---

## 0. 文档定位

本文是**架构设计（Architecture Design）**，不是 DSL 语法规范，也不是引擎实现细节。

它回答的问题是：

- 系统由哪些**组件**构成，各自职责边界在哪里？
- 一条规则从"被定义"到"在多个运行时执行"，**数据如何流动**？
- UI 和 Server 各算一遍，**结果不一致时怎么办**（裁决）？
- Function 不能随便写代码，那**治理流程**长什么样？
- 规则会变，**版本化和灰度**怎么保证不出事？

DSL 的字段级 schema、表达式文法、引擎内部实现，留给后续专门文档。

---

## 1. 设计目标与约束（回顾 + 细化）

### 1.1 核心目标
> **One Rule → Multiple Runtimes**：同一份规则 IR，在 UI / 微服务 / 第三方 UI 上一致执行。

### 1.2 架构必须满足的硬约束

| 编号 | 约束 | 含义 |
|------|------|------|
| C1 | **规则与实现解耦** | DSL 只描述规则，不含命令式实现代码 |
| C2 | **一致性优先** | 同一规则、同一输入，UI 与 Server 必须得到相同结果（或可解释的差异） |
| C3 | **Server 最终裁决** | UI 结果仅供体验，Server 结果具有法律/业务效力 |
| C4 | **Function 受治理** | 命名函数必须注册、版本化、双端实现对齐 |
| C5 | **可版本化 / 可回滚** | 任意时刻可定位"哪条交易用了哪个版本的规则" |
| C6 | **运行时无关** | 规则不依赖任何具体 UI 技术栈或服务框架 |

### 1.3 非目标（Out of Scope，本期不做）
- 不做可视化规则编辑器（后续产品化再议）
- 不做规则的自动机器学习/推荐
- 不替代工作流引擎（BPM）；DSL 只管"规则"，不管"长流程编排"

## 2. 分层架构总览

```
┌──────────────────────────────────────────────────────────────┐
│                      Authoring & Governance                    │
│   规则作者 → 规则编辑 → 校验/Lint → 审批 → 发布 → 版本仓库      │
└───────────────────────────────┬──────────────────────────────┘
                                 │ 发布(immutable bundle)
                                 ▼
┌──────────────────────────────────────────────────────────────┐
│                    Rule Distribution Layer                     │
│   Rule Repository (版本化) + Rule Bundle API + CDN/缓存        │
└───────┬───────────────────────┬───────────────────────┬───────┘
        │ pull bundle           │ pull bundle           │ pull bundle
        ▼                       ▼                       ▼
┌───────────────┐      ┌───────────────────┐   ┌──────────────────┐
│  UI Runtime   │      │   MS Runtime      │   │ 3rd-party UI     │
│ (实时,体验向)  │      │ (权威,最终裁决)    │   │ Runtime          │
├───────────────┤      ├───────────────────┤   ├──────────────────┤
│        共享同一个 Rule Execution Core (规则执行内核)            │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ Expression Engine │ Function Registry │ Pipeline Engine   │ │
│  │ + Scope Resolver(父子/跨节点) + Trigger Dispatcher        │ │
│  └──────────────────────────────────────────────────────────┘ │
└───────────────┘      └───────────────────┘   └──────────────────┘
                                 │
                                 ▼
                       ┌───────────────────┐
                       │  裁决/审计结果回写  │
                       │ (Server 为权威源)  │
                       └───────────────────┘
```

**三个层次：**

1. **Authoring & Governance（编写与治理层）**——规则的"出厂质检线"。
2. **Rule Distribution（分发层）**——版本化仓库 + 不可变 bundle 分发。
3. **Runtime（执行层）**——多个运行时，但共享**同一套执行内核**（这是一致性的根本保证）。

## 3. 核心组件设计

### 3.1 Rule Execution Core（规则执行内核）—— 一致性的基石

> **设计要点：内核必须是"同一份逻辑"，而不是"两份等价实现"。**

一致性问题（C2）的根因，是 UI 用 JS、Server 用 Java/Go，各写一遍校验逻辑导致 drift。架构上的根治办法是：

**内核只实现一次，编译/打包成多目标产物，而不是各端各写一遍。**

推荐方案（按优先级）：

| 方案 | 描述 | 一致性保证 | 代价 |
|------|------|-----------|------|
| **A. 单源多目标** ✅推荐 | 内核用一种语言写（如 TS/Rust），编译到 JS（UI）+ WASM 或原生（Server） | 最强：同一份代码 | 需要构建管线 |
| B. 共享规范 + 一致性测试 | 双端各实现，但用同一套**黄金测试集**（同输入同输出）做 CI 门禁 | 中：靠测试兜底 | 维护两套实现 |
| C. Server 计算、UI 仅展示 | UI 不算，所有计算走 Server | 最强但体验差 | 失去实时性，违背初衷 |

**建议：A 为主，B 为兜底回归测试。** 这是整个架构里最关键的决策——它把"前后端一致"从"靠纪律"变成"靠工程机制"。

---

#### ⚠️ 重要澄清："单源多目标"不是"把 DSL 翻译成多种语言"

这一点极易被误解，必须讲清楚，否则团队后续会反复争论。先区分系统里**两个不同的"单源"**：

| | 是什么 | 谁来写 | 是否单源 |
|---|---|---|---|
| **① 规则（DSL）** | 业务规则本身的 JSON，如 `amount * rate`、`sum(children.amount) <= parent.limit` | 业务/规则作者 | 天然单源（一份 JSON 分发各端，本项目初衷） |
| **② 内核（Engine）** | **读懂并执行**这些 JSON 的解释器代码 | 平台/中间件团队 | **ADR-1 讨论的就是它** |

**ADR-1 的"单源多目标"指的是 ②（内核），不是 ①（规则）。** 而"执行规则"在技术上有两条路线，常被混淆：

##### 路线 A（被否决）：代码生成 / 转译 —— "把每条 DSL 规则编译成各端源码"

```
amount * rate  ──codegen──→  UI:     function r1(c){ return c.amount * c.rate }   // JS
               ──codegen──→  Server: double r1(Ctx c){ return c.amount*c.rate; }  // Java
```

- 每个目标语言需要一个独立的代码生成器（codegen）。
- **致命问题：两个 codegen 对边界情况（四舍五入、空值、整数溢出、精度）的处理可能不同 → drift 重新出现。**
- 一致性问题并没有被根治，只是从"人手写两份"变成"两个生成器各生成一份"，隐患换了个位置。

##### 路线 B（✅ ADR-1 采用）：单一解释器 + 多平台部署

**内核（解释器）只写一份源码**，编译/打包成多个平台的可执行产物；规则 JSON 在运行时被它解释执行：

```
              内核源码（只写一次，如 TS 或 Rust）
                        │  编译 / 打包
              ┌─────────┴──────────┐
              ▼                    ▼
         JS bundle            WASM / 原生库
        （浏览器里跑）          （Server 里跑）
              │                    │
              └── 都加载同一份规则 JSON，运行时解释执行 ──┘
```

- **"多目标" = 同一份内核源码，产出多个平台的运行产物**（JS 给浏览器、WASM/原生给 Server）——**指部署平台，不指语言翻译**。
- 因为求值逻辑是**同一份编译产物**，边界行为（四舍五入、空值等）天然一致，这才是"靠工程机制根治 drift"。

##### 一句话区分

> ADR-1 **不是**"用工具把 DSL 生成多种语言的代码"（那是路线 A，会留下两个生成器导致 drift 的隐患）；
> 而是"**执行 DSL 的引擎只写一份源码，编译成多个平台的运行产物**，所有端加载同一份引擎产物去解释同一份规则"。

> **类比 SQL**：你不会把每条 SQL 翻译成 JS 和 Java 各跑一遍；你是让**同一个数据库引擎**到处部署，所有客户端发同样的 SQL 给它执行。DSL 内核扮演的就是"数据库引擎"这个角色——规则 JSON 就是 SQL。

---

内核内部由 5 个子模块组成：

#### 3.1.1 Expression Engine（表达式引擎）
- 解析并求值声明式表达式，如 `amount * rate`、`charge <= maxCharge`。
- **纯函数、无副作用、确定性**（同输入必同输出，禁止读时钟/随机/IO）。
- 支持内置算子、内置聚合（`sum/min/max/count`）、字段路径引用。

#### 3.1.2 Function Registry（函数注册表）
- 管理所有命名 Function（如 `calcFee`）的**元数据 + 双端实现绑定 + 版本**。
- 运行时按 `(functionId, version)` 解析到具体实现。
- 详见 §6 治理。

#### 3.1.3 Pipeline Engine（流水线引擎）
- 按 `steps` 顺序执行多步计算，前一步输出作为后一步输入上下文。
- 每个 step 可以是 Function 或子表达式。
- 支持中间结果可观测（用于调试/审计）。

#### 3.1.4 Scope Resolver（作用域解析器）—— 处理父子/跨节点
- 解析 `parent.limit`、`children.amount`、`sum(children.amount)` 这类**跨节点引用**。
- 维护交易树（主记录 + 子记录集合）的求值上下文。
- 详见 §5。

#### 3.1.5 Trigger Dispatcher（触发调度器）
- 决定每条规则**何时**执行：`on-change` / `before-calc` / `after-calc` / `on-submit`。
- UI 与 Server 用同一套 trigger 语义，避免"UI 算了 Server 没算"。

### 3.2 Rule Repository（规则仓库）
- 存储**不可变的规则版本**（immutable）。一经发布，内容不可改，只能发新版本。
- 元数据：`ruleSetId`、`version`、`status(draft/active/deprecated)`、`effectiveFrom`、`hash`、`author`、`approvedBy`。
- 提供 **Rule Bundle API**：运行时按 `ruleSetId@version` 拉取打包好的规则集（含其依赖的 Function 元数据）。

### 3.3 Runtime Adapters（运行时适配器）
- 每个运行时（UI / MS / 第三方）有一层薄适配器，负责：
  - 拉取并缓存 rule bundle；
  - 把本运行时的数据模型映射成内核要求的求值上下文；
  - 回收执行结果（校验错误、计算字段、审计轨迹）。
- **适配器是薄的，逻辑全在内核**——这是 C6（运行时无关）的落地方式。

## 4. UI / Server 双执行模型 + 裁决机制

这是本文最核心的两节之一（对应 C2、C3）。

### 4.1 双执行的角色分工

| 维度 | UI Runtime | Server Runtime (MS) |
|------|-----------|--------------------|
| 目的 | 体验（实时反馈） | 权威（最终结果） |
| 时机 | 用户输入即时执行 | 提交时 / 关键节点执行 |
| 信任级别 | **不可信**（仅提示） | **可信**（裁决源） |
| 失败后果 | 提示用户，可被绕过 | 拒绝交易，不可绕过 |
| 数据范围 | 可能只有局部数据 | 拥有完整、权威数据 |

**核心原则（C3）：UI 结果永远是"建议"，Server 结果永远是"判决"。**
即使 UI 校验全过，Server 仍必须独立重算一遍——绝不信任 UI 传来的计算结果或"已校验"标记。

### 4.2 执行时序

```
用户操作
  │
  ▼
[UI Runtime] 拉取 ruleSet@v 缓存
  │  on-change/after-calc 触发
  ▼
实时校验 + 实时计算 → 即时反馈(红框/提示/自动填值)
  │
  │ 用户点击"提交"，UI 把【原始输入】发给 Server
  │ （注意：发原始输入，不发 UI 算出来的结果）
  ▼
[Server Runtime] 用【相同 ruleSet@v】独立重算
  │  before-calc → calc → after-calc → on-submit 全量执行
  ▼
┌─────────────┬──────────────────────────────┐
│ 校验通过      │ 校验失败                       │
│ → 落库        │ → 拒绝 + 返回结构化错误         │
│ → 回写审计     │ → UI 高亮对应字段              │
└─────────────┴──────────────────────────────┘
```

**关键设计：提交时 UI 发送的是"原始输入数据"，而不是"UI 计算出的结果"。**
Server 从原始输入重新跑完整规则链，杜绝"前端篡改计算结果"。

### 4.3 裁决机制（Arbitration）

当 UI 和 Server 结果**不一致**时，定义清晰的裁决策略：

| 场景 | 裁决规则 |
|------|---------|
| 计算结果不一致 | **Server 值为准**，覆盖 UI 显示值，并记录差异到审计日志 |
| 校验结论不一致（UI 过 / Server 不过） | **以 Server 为准**，拒绝交易，返回 Server 的错误 |
| 校验结论不一致（UI 不过 / Server 过） | 通常是 UI 数据不全；**以 Server 为准**放行，但**告警**（说明 UI 规则可能过严或数据缺失） |
| 规则版本不一致（UI 用旧版，Server 用新版） | 以 Server 版本为准；若差异过大，提示用户刷新页面重载规则 |

**差异（divergence）必须被观测，不能静默。**
每次 Server 裁决覆盖 UI 结果，都应产生一条 `divergence event`：

```
divergence event = {
  txId, ruleId, field,
  uiValue, serverValue,
  uiVersion, serverVersion,
  reason: "calc-mismatch" | "validation-mismatch" | "version-skew"
}
```

这些事件汇总后是**衡量"前后端一致性健康度"的核心指标**——理想状态下趋近于 0；非 0 说明内核出现 drift 或数据上下文不一致，需要排查。

### 4.4 版本偏移（Version Skew）处理
- UI 长时间不刷新会用到旧规则版本。
- 提交请求里携带 `uiRuleVersion`；Server 比对自身 `activeVersion`。
- 偏移在容忍窗口内 → 正常裁决；超出 → 返回 `RULE_VERSION_STALE`，前端重载 bundle 后重试。

## 5. 父子作用域 / 跨节点执行模型

对应"复杂交易结构"需求（主记录 + 子记录 + 父子规则 + 跨节点）。

### 5.1 数据模型：交易树

```
Transaction (root / 主记录)
 ├─ field: limit, status, ...
 └─ children: [ child1, child2, ... ]   (子记录集合)
       └─ field: amount, rate, ...
```

### 5.2 引用语义（Scope Resolver 的职责）

| 引用形式 | 含义 | 求值位置 |
|---------|------|---------|
| `amount` | 当前节点字段 | 当前作用域 |
| `parent.limit` | 父节点字段 | 向上一层 |
| `children.amount` | 子集合的某字段（向量） | 向下展开 |
| `sum(children.amount)` | 子集合聚合 | 聚合后回到父作用域 |
| `root.xxx` | 主记录字段 | 顶层 |

### 5.3 求值上下文（Evaluation Context）
每条规则执行时，Scope Resolver 构造一个上下文：
- `self`：当前节点
- `parent`：直接父节点（可空）
- `root`：主记录
- `children`：当前节点的子集合（可空）

规则不直接拿数据库句柄——**只拿这个上下文**，从而保持纯函数特性（C1）。

### 5.4 求值顺序（依赖驱动）
- 子记录的计算字段，必须在父记录聚合校验（如 `sum(children.amount) <= parent.limit`）**之前**完成。
- 引擎构建**字段依赖图（DAG）**，拓扑排序后执行；检测到环则在 Lint 阶段报错（见 §6.4）。

```
child.charge = child.amount * child.rate     (先算每个子节点)
        ↓
parent.total = sum(children.charge)          (再聚合到父)
        ↓
validate: parent.total <= parent.limit       (最后校验)
```

## 6. Function 治理流程

对应 C4。Function 是 DSL 里唯一可以承载"复杂命令式逻辑"的地方，因此**必须重治理**，否则会成为绕过 DSL 解耦原则的后门。

### 6.1 为什么 Function 必须治理
- 表达式是受限、确定、可静态分析的；Function 是"黑盒代码"。
- 如果任由内联任意代码，就回到了"UI 一套 / Server 一套"的老路。
- 所以 Function 不允许在规则里写实现，**只允许"按 ID 引用已注册的函数"**。

### 6.2 Function 注册表条目

```
FunctionDescriptor {
  functionId:   "calcFee",
  version:      "1.3.0",
  signature:    { params: {...}, returns: {...} },   // 强类型契约
  purity:       "pure",            // 必须纯函数，无 IO/无副作用
  determinism:  "deterministic",   // 同输入同输出
  implementations: {
    ui:     <绑定: 单源编译产物 / 或 已对齐的 JS 实现>,
    server: <绑定: 单源编译产物 / 或 已对齐的实现>
  },
  ownership:    { team, approver },
  status:       "active | deprecated",
  goldenTests:  [ {input, expectedOutput}, ... ]   // 一致性回归集
}
```

### 6.3 治理生命周期

```
提案(Propose)
  → 定义签名/契约 + 双端实现 + goldenTests
  → 评审(签名是否纯、是否确定、是否双端对齐)
  → 一致性门禁(CI: UI实现 vs Server实现 跑 goldenTests，必须逐条相等)
  → 审批(Approve, 指定 owner)
  → 注册(分配 functionId@version, 不可变)
  → 激活(Active)
  → 废弃(Deprecate, 标记但保留以支持历史交易回放)
```

**一致性门禁是治理的核心**：任何 Function 在合入前，UI 实现与 Server 实现必须对同一组 goldenTests 产出**逐条相等**的结果，否则拒绝合入。这把 C4 落到了 CI 上。

### 6.4 规则集发布前的 Lint / 静态校验
规则在发布前要过静态检查（Authoring 层的"质检"）：
- 引用的 `functionId@version` 是否存在且 active；
- 字段路径是否合法（父子引用是否越界）；
- 依赖图是否有环；
- 表达式是否含禁用算子（IO/时钟/随机）；
- trigger 是否自洽（`after-calc` 校验是否真的有对应计算产出）。

## 7. 规则版本化与生命周期

对应 C5。

### 7.1 版本模型
- **规则集（RuleSet）按版本不可变发布**：`ruleSetId@version`，内容 + hash 固定。
- 每个版本含：规则列表、所引用 Function 的精确版本、生效时间窗。
- 交易落库时，**记录所用 `ruleSetId@version`**——保证任意历史交易可"按当时规则回放"。

### 7.2 发布与灰度
```
draft → (审批+lint通过) → staged → canary(灰度比例) → active → deprecated → archived
```
- **Canary**：新版本先对一小撮流量生效，监控 §4.3 的 divergence 指标，无异常再全量。
- **回滚**：把 active 指针切回上一个稳定版本即可（因版本不可变，回滚是 O(1) 的指针操作）。

### 7.3 生效时间与一致性
- UI 和 Server 必须解析到**同一个 active 版本**。
- 通过中心化的 "active version pointer"（每个 ruleSetId 一个）保证两端看到同一版本；UI 拉取时带回该 pointer，提交时回传给 Server 比对（§4.4）。

---

## 8. 端到端数据流（一次完整交易）

```
1. UI 启动 → 拉取 ruleSet@vN (active) → 本地缓存
2. 用户编辑主/子记录
3. UI 内核: Scope Resolver 建上下文 → 依赖DAG → Expression/Pipeline/Function 执行
   → 实时计算填值 + 实时校验提示
4. 用户提交 → UI 发送 { 原始输入树, uiRuleVersion=vN }
5. Server 适配器 → 拉/取本地 ruleSet@vN_active
6. Server 内核: 用相同逻辑独立重算全量规则
7. 版本比对(§4.4) + 裁决(§4.3)
8a. 通过 → 落库(含 ruleSet@version) + 回写权威计算值 + 审计
8b. 失败 → 结构化错误返回 → UI 高亮字段
9. 若 Server 值 != UI 值 → 产生 divergence event → 一致性监控
```

## 9. 非功能性需求

| 维度 | 要求 | 架构手段 |
|------|------|---------|
| 一致性 | UI/Server 同输入同输出 | 单源多目标内核(§3.1) + goldenTests + divergence 监控 |
| 性能(UI) | 实时校验 < 数十 ms | 内核纯计算、无 IO；规则 bundle 本地缓存 |
| 性能(Server) | 提交校验可控延迟 | 依赖 DAG 增量求值；Function 纯函数可缓存 |
| 安全 | 防前端篡改 | Server 从原始输入重算，不信任 UI 结果(§4.2) |
| 可审计 | 可追溯哪笔交易用哪版规则 | 交易记录 ruleSet@version；divergence/审计事件 |
| 可观测 | 一致性健康度可量化 | divergence event 指标化、告警 |
| 可回滚 | 出问题快速恢复 | 版本不可变 + active 指针切换 |

## 10. 关键架构决策记录（ADR 摘要）

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| ADR-1 | 如何保证前后端一致 | **单源多目标内核**：内核(解释器)写一份，编译成多平台产物（路线B），**而非把规则转译成多语言源码（路线A）** | 把一致性从"靠纪律"变"靠工程"，根治 drift。详见 §3.1 重要澄清 |
| ADR-2 | UI 提交什么 | **原始输入，非计算结果** | 防篡改，Server 独立裁决 |
| ADR-3 | 结果冲突谁说了算 | **Server 权威**，差异留痕 | 对应 C3，且可观测一致性 |
| ADR-4 | Function 如何防滥用 | **注册 + 双端对齐 + CI 门禁** | 堵住"内联代码"后门 |
| ADR-5 | 规则如何演进 | **不可变版本 + canary + 指针回滚** | 安全演进、可回放历史 |
| ADR-6 | 复杂逻辑边界 | 表达式→Formula→Function→Pipeline 升级 | 简单用表达式，复杂才上结构化 |

## 12. 小结

本架构的核心立意可以浓缩为三句话：

1. **一致性靠"同一份内核"，不靠"两份等价实现"**（ADR-1）。
2. **UI 求快只作建议，Server 求准独立裁决，差异必留痕**（§4）。
3. **Function 是受治理的后门，必须注册、对齐、CI 门禁**（§6）。

其余（父子作用域、版本化、灰度回滚）都是围绕这三条核心展开的工程支撑。
